# Отчёт по домашнему заданию: AI-инструкции для проекта CookHub

## Введение

Данное домашнее задание посвящено созданию и настройке AI-инструкций для автоматизации работы с кодом проекта CookHub (https://github.com/kovalenkojuls/otus-java-graduation-work). В работе использованы инструменты, связанные с AI-ассистентом в редакторе кода: **Continue (через `.continuerc.json` и `.continue/rules.md`)**, **Ollama (Qwen 3 Coder Next)** и интеграция в **JetBrains IDE (IDEA)**.

## Используемые инструменты

| Инструмент | Назначение                                                                                                                             |
| :--- |:---------------------------------------------------------------------------------------------------------------------------------------|
| **Continue** | Расширение для редакторов кода (IDEA, VS Code), позволяющее использовать LLM для работы с кодом. Управляется через `.continuerc.json`. |
| **Ollama + Kwen 3 Coder Next** | LLM-провайдер + `Qwen 3 Coder Next` — специализированная модель для программирования. Она вызывается через Ollama из Continue.         |
| **JetBrains IDEA** | Основная среда разработки, где используется Continue для взаимодействия с кодом.                                                       |

## Описание структуры AI-документации

### `.continuerc.json`
**Назначение**: Файл конфигурации для Continue. Здесь указываются настройки подключения к LLM, а также глобальные правила поведения агента.

**Что включает**:
-   Имя модели
-   Температуру (`temperature: 0.2` — для минимизации креативности и максимизации точности).
-   Список правил (`rules`), которые агент обязан соблюдать (ссылки на контекстные файлы).

### `.continue/rules.md`
**Назначение**: Персональные правила поведения агента. Это "своеобразный свод законов", который описывает, как агент должен взаимодействовать с проектом. Здесь фиксируются принципы архитектуры (микросервисы, Kafka), требования к безопасности (всегда валидация), подходы к коду (TDD, самодокументируемость) и др.

**Что включает**:
-   Общие правила (контекст, безопасность, консистентность).
-   Правила для каждого микросервиса (Web, Email, Telegram).
-   Принципы написания кода (стиль, комментарии, тесты).
-   Работа с задачами и деплоем.

### `docs/ai/CONTEXT.md`
**Назначение**: Базовый контекст для AI-агента. Здесь описывается суть проекта, его цели, пользователи, технологический стек и, самое главное, **актуальный контекст для агента** — что он должен учитывать при работе.

**Что включает**:
-   Описание платформы и её микросервисов.
-   Ключевые пользователи.
-   Технологический стек.
-   Актуальные требования для агента (распределенность, тестирование, стиль кода).

### `docs/ai/ARCHITECTURE.md`
**Назначение**: Подробная техническая архитектура проекта. Это "карта", которая показывает, как все части проекта взаимодействуют друг с другом.

**Что включает**:
-   Общую схему микросервисов и потоки данных (например, регистрация -> Kafka -> Email Service).
-   Ответственность каждого микросервиса (Web, Email, Telegram Bot).
-   Потоки данных (регистрация, просмотр рецептов).
-   Принципы проектирования (асинхронная коммуникация, слабая связанность).

### `docs/ai/TASKS.md`
**Назначение**: Коллекция примеров типовых задач, которые агент должен уметь выполнять. Это помогает ему понимать, как правильно формулировать ответы на запросы и какие подходы использовать.

**Что включает**:
-   5 категорий задач (добавление категории, обработка ошибок email, новая команда в Telegram, тесты, пагинация).
-   Примеры формулировок для запросов к агенту (примеры "промптов").

## Описание выполненных действий

### Шаг 1: Создание структуры AI-документации

В корне проекта были созданы следующие файлы и папки:

-   `.continuerc.json` — конфигурация Continue.
-   `.continue/rules.md` — правила поведения агента.
-   `docs/ai/` — папка с контекстной документацией:
    -   `CONTEXT.md` — общий контекст проекта.
    -   `ARCHITECTURE.md` — детальная архитектура микросервисов.
    -   `TASKS.md` — примеры типовых задач и запросов.

### Шаг 2: Формулирование правил и контекста

Во всех файлах были описаны правила, которые учитывают:

-   **Микросервисную архитектуру**: Изменения в одном сервисе (например, Web Application) могут повлиять на другие (Email Service через Kafka).
-   **Безопасность**: Всегда валидация форм, защита от CSRF/XSS, проверка ролей для доступа к админке.
-   **Согласованность**: Соблюдение стиля кода (например, `@Service` + `@Repository` + `@Controller`).
-   **Качество**: Обязательные тесты (юнит и интеграционные) для каждого изменения.

### Шаг 3: Практическая проверка

#### Что хотели, чтобы агент внёс:

Реализовать **повторную попытку отправки email** в микросервисе `cookhub-email-service`, если SMTP-сервер недоступен.

#### Промт (запрос к агенту):

> "Реализуй повторную попытку отправки email в `EmailService`, если SMTP-сервер недоступен. Используй Kafka DLQ для неуспешных попыток. Следуй правилам из `.continue/rules.md`, в частности, пункта про Email Service."

#### Результат проверки:

Агент предложил решение:

-   **Обработка исключений**: Добавлен `try-catch` блок в метод отправки email.
-   **Повторные попытки**: Реализована простая механика повторов (например, 3 попытки с паузой).
-   **Kafka DLQ**: При неудаче сообщение отправляется в отдельный топик `registration-emails-dlq`.
-   **Соответствие правилам**: Использованы классы `EmailListener`, `EmailDto`, соблюдена структура проекта (service -> dto).

Вывод: Агент корректно применил правила из `ARCHITECTURE.md` и `.continue/rules.md`, не нарушил микросервисную границу (не стал копировать код из Web Application) и предложил технически обоснованное решение.
