# Контекст проекта CookHub

CookHub — это платформа для обмена кулинарными рецептами, состоящая из веб-приложения и вспомогательных микросервисов. Проект разработан как выпускная квалификационная работа (graduates' project) и демонстрирует применение современных подходов к разработке распределенных систем: микросервисной архитектуры, асинхронной коммуникации через Kafka, аутентификации, пагинации, фильтрации и интеграции с внешними сервисами (Telegram).

## Основные цели и задачи

-   **Создать удобную платформу**: Позволить пользователям находить, делиться и сохранять кулинарные рецепты.
-   **Обеспечить гибкость и масштабируемость**: Использовать микросервисы для разделения ответственности и возможности независимого масштабирования компонентов.
-   **Продемонстрировать технические навыки**: Использование Spring Boot, Spring Security, Kafka, PostgreSQL, Docker, и интеграции с Telegram API.

## Ключевые пользователи

-   **Обычные пользователи**: Видят список рецептов, создают свои, ставят лайки, подписываются на авторов.
-   **Администраторы**: Управляют списком пользователей (только через админ-панель веб-приложения).
-   **Пользователи Telegram**: Получают случайные рецепты через бота.

## Технологический стек

-   **Backend (микросервисы)**: Java, Spring Boot, Hibernate, Spring Security, Spring Session, Kafka.
-   **Frontend (веб-приложение)**: Thymeleaf, Bootstrap, JavaScript.
-   **База данных**: PostgreSQL.
-   **Деплой**: Docker, Docker Compose (через `docker/` и скрипт `start.sh`).
-   **Инструменты**: reCAPTCHA, Apache Kafka, Zookeeper, Kafka UI.

## Краткое описание микросервисов

-   **CookHub Web Application**: Главное веб-приложение с UI, авторизацией и управлением рецептами.
-   **CookHub Email Service**: Микросервис для асинхронной отправки email-уведомлений через Kafka.
-   **CookHub Telegram Bot**: Бот для Telegram, предоставляющий базовые функции (показ случайных рецептов).

## Актуальный контекст для ИИ-агента

При работе с проектом нужно:

-   Учитывать распределенную природу системы. Изменения в одном микросервисе могут влиять на другие через Kafka.
-   Сохранять согласованность принципов: единообразие в валидации, управлении сессиями, безопасности.
-   Следить за тем, чтобы новые фичи соответствовали заявленной архитектуре и принципам (например, автоматическая генерация UI по Enum).
-   Все изменения должны тестироваться (юнит и интеграционно).
-   Код должен соответствовать стилю и структуре, принятой в каждом из микросервисов (web, email, telegram).
