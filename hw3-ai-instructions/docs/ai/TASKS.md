# Примеры типовых задач для CookHub

## Общие принципы

-   **Контекст**: Все задачи должны рассматриваться в контексте микросервисной архитектуры и технологического стека.
-   **Качество**: Все изменения проходят тестирование (юнит, интеграционное).
-   **Согласованность**: Изменения должны соответствовать принятым стандартам кодирования и архитектурным решениям (например, генерация UI по Enum, валидация на сервере).

## Категории задач

### 1. Добавление новой категории рецепта (Web Application)

-   **Задача**: Добавить новую категорию (например, `DESSERT`) в Enum, описывающий категории.
-   **Ожидаемый результат**:
    -   Список рецептов на UI автоматически отображает новую категорию.
    -   Формы создания/редактирования рецептов автоматически содержат новую категорию.
    -   База данных не требует миграции, если тип хранения позволяет это (varchar + валидация на уровне приложения).
-   **Риски**: Проверить, что в Enum корректно прописаны значения для сохранения в БД и отображения в UI.

### 2. Обработка ошибки при отправке email (Email Service)

-   **Задача**: Реализовать повторную попытку отправки email при временных ошибках (например, недоступность SMTP-сервера).
-   **Ожидаемый результат**:
    -   Сообщения, вызвавшие ошибку, не теряются.
    -   Используется механика повторов (например, с экспоненциальной задержкой).
    -   Ошибки логируются и попадают в систему мониторинга (например, в Kafka Dead Letter Queue).
-   **Риски**: Не создать бесконечный цикл повторов. Не навалить нагрузку на SMTP-сервер.

### 3. Добавление нового запроса в Telegram Bot (Telegram Bot)

-   **Задача**: Реализовать команду `/favorites`, которая отправляет пользователю его избранные рецепты.
-   **Ожидаемый результат**:
    -   Бот запрашивает у пользователя (через команду) его идентификатор (или использует `from.id` из Telegram).
    -   Бот получает список избранных рецептов.
    -   Бот отправляет сообщение с кратким списком (например, "Рецепт 1: Название, Автор".
-   **Риски**: Как получить данные? Через REST API Web Application или напрямую к БД? (Скорее всего, через API).

### 4. Написание интеграционного теста для формы фильтрации (Web Application)

-   **Задача**: Написать тест, который имитирует отправку формы фильтрации по автору.
-   **Ожидаемый результат**:
    -   Тест проверяет, что в ответ приходит только список рецептов заданного автора.
    -   Проверка валидации (например, пустой автор возвращает ошибку).
-   **Инструменты**: Spring Boot Test, Testcontainers для PostgreSQL, MockMvc.

### 5. Оптимизация запросов к БД (Web Application)

-   **Задача**: Реализовать пагинацию для списков рецептов (для большого объема данных).
-   **Ожидаемый результат**:
    -   Список рецептов отображается по частям (страницы).
    -   Используется `Pageable` и `Page` из Spring Data JPA.
    -   Тесты на корректность расчета общего количества страниц и элементов.
-   **Риски**: Избежать N+1 проблемы при загрузке связанных данных.
