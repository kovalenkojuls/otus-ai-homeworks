# Правила поведения агента для проекта CookHub

## Общие правила

1.  **Контекст всегда в голове**: При любой задаче помни про архитектуру (микросервисы: Web, Email, Telegram), технологии (Spring Boot, Kafka, PostgreSQL) и принципы (асинхронная коммуникация, разделение ответственности).
2.  **Безопасность первая**: Любые изменения в веб-приложении должны соответствовать политике Spring Security (валидация, защита от CSRF, XSS). Никогда не добавляй SQL-инъекции.
3.  **Консистентность**: Следуй существующим стилям кода и соглашениям. Если в проекте используется `@Service` + `@Repository` + `@Controller`, не создавай альтернативные подходы без обоснования.
4.  **Микросервисы не врут**: Код для одного микросервиса нельзя просто так брать и вставлять в другой. Контекст и зависимости могут отличаться.
5.  **Данные в Kafka**: При добавлении новых событий в Kafka (например, `registration-emails`) убедись, что потребитель (consumer) готов их обработать, иначе сообщения будут теряться или накапливаться.

## Правила для каждого микросервиса

### Web Application

-   Все формы валидируются на сервере ( Hibernate Validator).
-   UI генерируется автоматически по Enum категорий рецептов.
-   Фильтрация и пагинация обязательны для списков.
-   Доступ к редактированию рецепта проверяется по автору.
-   Админка доступна только с ролью `ADMIN`.

### Email Service

-   Каждое письмо имеет свой шаблон и обработчик.
-   Ошибки отправки логируются и попадают в DLQ.
-   Не блокируй Kafka-топик на долго (оптимизируй работу с SMTP).

### Telegram Bot

-   Ответы на команды должны быть быстрыми (максимум 5 секунд).
-   Рецепты отправляются в краткой форме (заголовок, категория, ссылка на полный текст).
-   Обрабатывай отсутствие данных (если нет рецептов по категории).

## Принципы написания кода

-   Код должен быть самодокументируемым (хорошие имена переменных, методов).
-   Комментарии пиши только для сложной бизнес-логики или нетривиальных решений.
-   Тесты пиши перед кодом (TDD) или параллельно с ним.
-   Используй `final` для констант, `var` только там, где тип очевиден.

## Работа с задачами

-   Дели задачи на маленькие части (примеры в `ai/TASKS.md`).
-   Прежде чем менять код, подумай: как это повлияет на другие микросервисы?
-   Не бойся задавать вопросы (если они были бы написаны здесь, они были бы в `CONTEXT.md` или `ARCHITECTURE.md`).

## Деплой и окружение

-   Используй `start.sh` для развертывания.
-   Переменные окружения не хранить в коде.
-   Логи пиши в консоль (Docker будет их собирать).
-   Перед пушем запускай тесты: `mvn test` (примеры `pom.xml` в каждом сервисе).
